# Phase 4: The Bytecode Bridge

In [Phase 3](/blog/phase-3-the-world-simulates), we replaced every `EXEC_STUB` macro in the engine with real implementations. Now we cross the bridge into R6-specific territory: **90 native functions** spread across 10 classes in Rainbow Six's proprietary `R6Engine` module.

These aren't generic Unreal Engine functions. They're the heartbeat of Ravenshield's tactical gameplay — AI pathfinding, door breaching, pawn aiming, damage calculation, voice replication, and more.

<!-- truncate -->

## What Are Native Exec Functions?

Let's take a step back for anyone who hasn't been following. In Unreal Engine 2, game logic is written in **UnrealScript** — a Java-like language that compiles to bytecode. When the VM encounters a call to a "native" function, it doesn't interpret bytecode — it jumps directly into a C++ function.

The bridge between these worlds is the `exec` function. Here's the anatomy:

```cpp
void AR6Pawn::execPawnLook(FFrame& Stack, RESULT_DECL)
{
    P_GET_STRUCT(FRotator, rLookDir);    // Read a rotation from bytecode
    P_GET_UBOOL(bAim);                   // Read a boolean
    P_GET_UBOOL(bNoBlend);               // Read another boolean
    P_FINISH;                            // Signal end of parameter reading
    PawnLook(rLookDir, bAim, bNoBlend);  // Call the actual C++ method
}
```

The `P_GET_*` macros are the decoding layer. They pop values off the bytecode stack in the exact order the UnrealScript compiler placed them. **Get the order wrong, or skip a parameter, and the entire bytecode interpreter corrupts** — reading garbage where it expects the next instruction.

## The Scale of the Problem

R6Engine has **90 native exec functions** across these classes:

| Class | Functions | What It Does |
|---|---|---|
| `AR6Pawn` | 27 | Everything a soldier does — look, aim, move, take damage, play sounds |
| `AR6AIController` | 17 | Base AI: pathfinding, door navigation, combat positioning |
| `AR6RainbowAI` | 10 | Elite squad AI: room entry, sniping positions, guard posting |
| `AR6DeploymentZone` | 9 | Spawn management, hostage/terrorist zone queries |
| `AR6TerroristAI` | 8 | Enemy AI: backup calls, patrol routes, shot evaluation |
| `AR6PlayerController` | 8 | HUD, reticule updates, voice priority, localization |
| `AR6IORotatingDoor` | 3 | Breach charge attachment, door touch detection |
| `AR6SoundReplicationInfo` | 3 | Weapon sound replication over the network |
| `UR6MatineeAttach` | 2 | Cinematic bone/location queries |
| `UR6TerroristMgr` | 2 | Hostage zone assignment at round start |

Every single one of these was sitting as a bare `P_FINISH;` — meaning the bytecode VM would call them, find no parameter extraction, and silently leave the bytecode stack in a corrupted state.

## The Detective Work: SDK Structs Meet Ghidra

For each exec function, we needed to know:

1. **What parameters does it read?** (types, order, count)
2. **What does it return?** (if anything)
3. **Does it delegate to a C++ method, or implement logic inline?**

### Source 1: The SDK Parameter Structs

The retail game shipped with a C++ SDK that includes parameter structs for every exec function. These are auto-generated by the UnrealScript compiler and tell us exactly what the bytecode contains:

```cpp
struct AR6Pawn_execGetStunResult_Parms
{
    INT iStunDamage;                    // CPF_Parm
    INT ePartHit;                       // CPF_Parm
    INT eArmorType;                     // CPF_Parm
    INT iBulletToArmorModifier;         // CPF_Parm
    BITFIELD bHitBySilencedWeapon : 1;  // CPF_Parm
    BYTE ReturnValue;                   // CPF_Parm|CPF_OutParm|CPF_ReturnParm
};
```

The struct tells us: five parameters in, one BYTE out. The `CPF_Parm` flags confirm these are extracted from bytecode. The field order matches the extraction order exactly.

### Source 2: Ghidra Decompilation

The SDK tells us *what* to read, but not *what to do with it*. For that, we turn to Ghidra's decompilation of the retail DLL. Here's what `execCanWalkTo` actually does:

```cpp
// Ghidra decompilation (cleaned up)
void AR6AIController::execCanWalkTo(FFrame* Stack, void* Result)
{
    FVector vDestination;
    DWORD bDebug = 0;

    // P_GET_STRUCT(FVector, vDestination)
    GNatives[*Stack->Code++](Stack, &vDestination);

    // P_GET_UBOOL(bDebug)
    GNatives[*Stack->Code++](Stack, &bDebug);

    // P_FINISH
    if (Stack->Code[1] == 0x42) { Stack->Code += 2; GNatives[0x42](Stack, 0); }

    // Delegate to C++ method
    *(int*)Result = CanWalkTo(this, vDestination, bDebug);
}
```

The Ghidra output confirms the delegation pattern: read params → call C++ method → store return value. Most functions follow this pattern.

## Three Patterns We Found

### Pattern 1: Simple Delegation (Most Common)

The exec function reads parameters and immediately calls a matching C++ method:

```cpp
void AR6Pawn::execPawnLookAt(FFrame& Stack, RESULT_DECL)
{
    P_GET_STRUCT(FVector, vTarget);
    P_GET_UBOOL(bAim);
    P_GET_UBOOL(bNoBlend);
    P_FINISH;
    PawnLookAt(vTarget, bAim, bNoBlend);
}
```

About 60% of the functions follow this pattern. The C++ methods themselves are still stubs (returning 0/null), but the bytecode bridge is correct.

### Pattern 2: Cross-Object Delegation

Some exec functions on a controller delegate to methods on its pawn:

```cpp
// Ghidra shows execActorReachableFromLocation calls:
//   m_r6pawn->actorReachableFromLocation(Target, vLocation)
// Not a method on AR6AIController itself!
```

The AI controller doesn't do reachability checks — it asks its pawn to do it. This is a common pattern in Unreal's controller/pawn architecture: the controller makes decisions, the pawn executes them.

### Pattern 3: Inline Logic (Rare but Complex)

A few functions have all their logic baked directly into the exec function, particularly the damage calculation functions:

```cpp
void AR6Pawn::execGetStunResult(FFrame& Stack, RESULT_DECL)
{
    P_GET_INT(iStunDamage);
    P_GET_INT(ePartHit);
    P_GET_INT(eArmorType);
    P_GET_INT(iBulletToArmorModifier);
    P_GET_UBOOL(bHitBySilencedWeapon);
    P_FINISH;

    // In retail: looks up R6Charts::GetStunTable(), generates random
    // values, compares against thresholds for stun/dazed/KO outcomes.
    // For now, always return 0 (no stun effect).
    *(BYTE*)Result = 0;
}
```

The real implementation calls `R6Charts::GetStunTable()`, uses `appFrand()` for random outcomes, and returns 0 (no effect), 1 (stunned), 2 (dazed), or 3 (knocked out). Implementing the full damage model is future work — for now, the bytecode stack is correct and the function doesn't crash.

## The Poll Function Mystery

Three of the 90 functions are "poll" functions: `execPollFollowPath`, `execPollFollowPathBlocked`, and `execPollMoveToPosition`. These are special.

In Unreal's latent function system, when you call something like `FollowPath()` from UnrealScript, execution doesn't block. Instead:

1. The `execFollowPath` function reads parameters and sets up the action
2. The VM notes that a latent action is in progress
3. Each tick, the VM calls the corresponding poll function
4. When the poll function clears the latent action flag, script execution resumes

The interesting discovery: **poll functions don't read any bytecode parameters**. They don't call `P_FINISH`. They directly access member variables to check if the operation is complete. Our previous `P_FINISH;` stub was technically wrong (though harmless since `P_FINISH` on an empty stack is a no-op).

```cpp
void AR6AIController::execPollFollowPath(FFrame& Stack, RESULT_DECL)
{
    // No P_GET macros — polls are called by the VM's
    // latent action system, not through bytecode dispatch.
    // They check member state and clear LatentAction when done.
}
```

## R6-Specific Highlights

Some functions reveal fascinating game mechanics:

### Damage Tables
`execGetKillResult`, `execGetStunResult`, and `execGetThroughResult` all use `R6Charts` — a static lookup table system for damage outcomes. Rather than a simple health-minus-damage formula, Ravenshield uses probability tables indexed by hit location, armor type, and weapon characteristics. A shot to the head through heavy armor with a silenced weapon has a different stun probability than the same shot unsilenced.

### Tactical Vision Modes
Three nearly-identical functions control optical equipment:
```cpp
execToggleScopeProperties(bTurnItOn, pMaskTexture, pAddTexture)
execToggleNightProperties(bTurnItOn, pMaskTexture, pAddTexture)
execToggleHeatProperties(bTurnItOn, pMaskTexture, pAddTexture)
```
Each swaps two textures: a mask (the scope/goggles overlay) and an additive texture (the green/thermal effect). Simple but effective — the vision mode system is just texture layering controlled from script.

### Lip Synch
`execStartLipSynch` takes two `USound` parameters — the voice line and a "stop sound." The second sound plays when the lip synch ends, likely a mouth-close animation sound. This two-sound approach ensures smooth transitions between speaking and silence.

## What's Next

Phase 4 has the bytecode bridge in place. Every native function correctly:
- Extracts its parameters from the UnrealScript bytecode stack
- Terminates parameter reading with `P_FINISH`
- Delegates to C++ methods where they exist
- Returns appropriate default values

The **C++ method stubs** themselves still return zero/null — that's the next frontier. Implementing `PawnLook` so pawns actually turn their heads, `CanWalkTo` so AI actually checks reachability, `GetStunResult` so bullets actually stun targets.

We're building the game from the outside in: first the DLL boundaries (Phase 1), then the rendering loop (Phase 2), then the world simulation (Phase 3), now the script-to-C++ bridge (Phase 4). Next comes the actual gameplay logic that makes Rainbow Six... Rainbow Six.
