Some functions practically write themselves. Others are like reading ancient hieroglyphics with a broken decoder ring. This post is about the second kind — the 62 functions in `UnActor.cpp` tagged `IMPL_DIVERGE`, and what we learned by tracking each one back to its original machine code.

## What Even Is IMPL_DIVERGE?

Before diving in, a quick primer on the attribution macros we use.

Every function in this decompilation has to be labelled with one of three macros before its definition:

- **`IMPL_MATCH("Engine.dll", 0xADDRESS)`** — "I believe this matches the retail binary exactly at this virtual address."
- **`IMPL_EMPTY("reason")`** — "Retail is also an empty stub. Ghidra confirmed."
- **`IMPL_DIVERGE("reason")`** — "Our implementation differs from retail. Here's why."

This forces every function to be explicitly accounted for. There's no "I'll document this later" escape hatch — `IMPL_TODO` and `IMPL_APPROX` are banned macros that cause build failures.

`IMPL_DIVERGE` is the honest label. It says: "We know this differs from retail, and here's why." The *why* is what this post is about.

## The 62 Suspects

`UnActor.cpp` is massive — it implements `AActor`, the base class for every interactive object in the game. When we audited it, 62 functions carried `IMPL_DIVERGE` labels, many with vague reasons like *"too complex"* or *"uses binary-specific globals"*. Vague reasons are a problem: they tell future developers nothing useful.

Our goal: go through all 62, open the Ghidra decompilation of `Engine.dll`, understand *why* each one diverges, and write a precise reason.

## The Ghidra Workflow

The key tool is `ghidra/exports/Engine/_global.cpp` — a ~600,000-line file generated by Ghidra's decompiler from the retail `Engine.dll`. It's not perfect C++. Ghidra uses placeholder names like `DAT_10666f4` for global variables it couldn't identify, and variables are typed as `undefined4` when their type is unknown. But it gives you the *structure* of every function: what it reads, what it calls, what it writes.

For each IMPL_DIVERGE, the workflow was:

1. Find the function in `_global.cpp` (via the export name in comments)
2. Note the virtual address from the comment (e.g. `/* 0x127b40 5571 ?execStopAllMusic@AActor... */`)
3. Compare against our stub
4. Write a precise divergence reason: what the retail does that we don't

## Category 1: The Audio Subsystem Chain

Twelve functions fall into this category. They include `execStopAllMusic`, `execStopAllSoundsActor`, `execStopSound`, and similar audio control functions. Here's what the retail `execStopAllMusic` actually does (Ghidra at `0x127b40`):

```c
void execStopAllMusic(AActor *this, FFrame *param_1, void *param_2)
{
    // P_FINISH equivalent — advances bytecode
    iVar1 = *(int *)(param_1 + 0xc);
    pcVar3 = (char *)(iVar1 + 1);
    *(char **)(param_1 + 0xc) = pcVar3;
    if (*pcVar3 == 'B') {
        *(int *)(param_1 + 0xc) = iVar1 + 2;
        (**(code **)(GNatives_exref + 0x108))(param_1, 0);
    }
    // Then: navigate to audio subsystem and call it
    piVar2 = *(int **)(*(int *)(*(int *)(this + 0x328) + 0x44) + 0x48);
    if (piVar2 != NULL) {
        (**(code **)(*piVar2 + 0xfc))();  // audio vtable slot 0x3f
    }
}
```

The chain is: `this→XLevel (this+0x328)` → `XLevel+0x44` → `[ptr]+0x48` → audio subsystem vtable. Our stubs parse the parameters but don't call the audio system — we don't have it yet. Every one of these is a *genuine, permanent* divergence until the audio DLL is integrated.

## Category 2: The Binary-Specific Global Pattern

Several functions read from or write to hard-coded memory addresses like `DAT_10666f4`, `DAT_10793088`, etc. These are globals baked into the retail binary — they exist at fixed offsets in the loaded DLL but have no names in our source.

A good example is `execGetServerBeacon` (`0x1240c0`), which simply reads `DAT_10793088` (a cached server beacon string) and returns it. Our stub returns an empty string instead. The data exists in retail RAM at a specific address; we have no equivalent.

`AActor::PreNetReceive` (`0x1037e30`) is the most extreme example. It saves ~20 actor fields to addresses `DAT_106666f4` through `DAT_10666728` — a snapshot of the actor state before network updates arrive. `PostNetReceiveLocation` then reads those snapshots back to move the actor smoothly. This snapshot-then-apply pattern for network interpolation is entirely absent from our source because we don't have those address slots.

## Category 3: The FMemStack Arena Fix

Not everything was a divergence — we found one function where the stub was actually *wrong*. `AActor::CheckOwnerUpdated` (`0x103c3460`) builds a linked list of actors for replication. The original stub just allocated on the heap. But Ghidra shows it actually uses `GEngineMem.PushBytes(8, 8)` — a **memory arena** (stack allocator), not the heap.

What's a memory arena? Rather than calling `malloc`/`new` for each allocation, a stack allocator bumps a pointer through a pre-allocated block. This is dramatically faster for per-frame allocations — you allocate thousands of 8-byte nodes each frame, and at the end of the frame the entire arena is reset in one pointer assignment. No individual frees needed.

The fix was to change:
```cpp
// Wrong: heap allocation
TReplicationNode* Node = new TReplicationNode(this, ...);
```
to:
```cpp
// Correct: arena allocation matching Ghidra 0x103c3460
BYTE* Node = (BYTE*)GEngineMem.PushBytes(8, 8);
*(void**)Node       = (void*)this;
*(void**)(Node + 4) = (void*)*(void**)SomeHead;
```

This was already committed by the prior session but it's worth explaining *why* it matters — functional equivalence with the correct allocator is closer to true byte parity than just heap allocation.

## Category 4: Karma Physics

Eight functions related to Karma (the Havok-based physics engine) are diverged. Karma is a licensed middleware with its own opaque API. Our stubs call `KInitConstraintData`, `KDestroyConstraint`, etc., but the exact function signatures and data layout differ from retail. These will remain diverged until we reverse-engineer the Karma headers from Ghidra.

## Category 5: Complex One-Off Functions

Some functions just can't be reconstructed yet because they depend on structs we haven't fully mapped:

- **`DbgVectorAdd` (`0x103794d0`, 428 bytes)**: Manages a per-actor debug vector cache in `m_dbgVectorInfo`. The `FDbgVectorInfo` struct layout is only partially known.
- **`DbgVectorDraw` (`0x103791f0`, 681 bytes)**: Iterates that same cache and draws each entry as a cylinder + label via `FRenderInterface`. Requires render interface type mappings we don't have yet.
- **`ABrush::OldBuildCoords` (`0x10307930`, 471 bytes)**: Builds model-space coordinate frames from two `FScale` fields at offsets `+0x3B0` and `+0x3C4` in the `ABrush` object. Those field offsets haven't been confirmed in our `EngineClasses.h` yet.

## The Value of Precise Reasons

After all this, the IMPL_DIVERGE *count* didn't drop (still 62). But that's not really the point. The value is that each divergence is now *informative*:

Before:
```cpp
IMPL_DIVERGE("too complex, uses binary globals")
```

After:
```cpp
// Ghidra 0x103791f0 (681 bytes): iterates m_dbgVectorInfo entries and draws each as a
// cylinder + label via FRenderInterface. Requires render interface not fully mapped.
IMPL_DIVERGE("iterates m_dbgVectorInfo and draws via FRenderInterface; render types not mapped (Ghidra 0x103791f0)")
```

The future developer working on `FRenderInterface` can search for `IMPL_DIVERGE` references, find `DbgVectorDraw`, and know exactly what needs to be done. The Ghidra address is right there — no archaeological re-dig required.

That's the real win: turning a pile of vague TODOs into a roadmap.
