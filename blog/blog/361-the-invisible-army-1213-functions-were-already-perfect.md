## The Measurement Problem

In decompilation work, you live and die by your metrics. Our primary metric is **byte parity** — does the rebuilt DLL produce the exact same machine code as the retail binary? We'd been sitting at 338 passing functions for a while, and it felt like pulling teeth to get each new one over the line.

The problem wasn't our code quality. It was our *measurement coverage*.

When you write a function like `AR6Pawn::GetStatusOtherTeam()`, you put an `IMPL_MATCH` annotation before it that says "this function lives at address `0x10002b10` in the retail DLL." The parity tool then compares your compiled output against those specific retail bytes. No annotation, no check.

But here's the thing — C++ class macros generate a *lot* of code behind the scenes. Every class that uses Unreal's `DECLARE_CLASS` macro gets a `StaticClass()` method, an `InternalConstructor()`, an `operator new()` allocator, a destructor, and sometimes copy constructors and assignment operators. These are all compiled, linked, and exported — but because they're expanded from macros, nobody writes `IMPL_MATCH` annotations for them.

## The Hypothesis

What if those macro-generated functions are... already correct? After all, `StaticClass()` is literally just:

```cpp
static UClass* StaticClass() { return &PrivateStaticClass; }
```

Which compiles to exactly 6 bytes:
```
mov eax, [address_of_PrivateStaticClass]
ret
```

And `InternalConstructor()` is 14 bytes of placement-new boilerplate. These are so simple that even across different MSVC versions and compilation flags, the optimizer produces identical output.

I wrote a script (`find_unannotated_matches.py`) that:
1. Loads every exported function from our rebuilt DLLs
2. Finds the corresponding function in the retail DLLs (via Ghidra's function index)
3. Compares the bytes with relocation masking
4. Reports which unannotated functions are already byte-perfect

The result? **1,213 functions** were sitting there, perfect, unrecognised.

## The .parity Manifest System

We couldn't just add `IMPL_MATCH` directly to these functions — they're generated by macros deep inside Unreal's header files. Putting annotations there would be fragile and ugly.

Instead, I created a new file format: `.parity` manifests. These are simple text files that live alongside the source code and look like this:

```
IMPL_MATCH("R6Engine.dll", 0x10001830)
?StaticClass@AR6GameReplicationInfo@@SAPAVUClass@@XZ // StaticClass (6 bytes)

IMPL_MATCH("R6Engine.dll", 0x10002b10)
?StaticClass@AR6Pawn@@SAPAVUClass@@XZ // StaticClass (6 bytes)

IMPL_MATCH("R6Engine.dll", 0x10002b20)
??2AR6Pawn@@SAPAXIPAVUObject@@VFName@@K@Z // operator_new (44 bytes)
```

Each entry has the `IMPL_MATCH` annotation followed by the **mangled C++ name** — the same symbol the linker sees. This sidesteps all the complexities of parsing C++ declarations. The verify tool just looks up the mangled name directly in the `.map` file and finds exactly where our function lives.

The manifests are auto-generated by `gen_parity_manifests.py`, which runs the same byte comparison that originally discovered the matches. They're checked into source control and will evolve as the codebase changes.

## The Relocation Masking Trick

You might wonder: how can functions compiled at completely different addresses produce "identical" bytes? After all, `mov eax, [0x10001234]` and `mov eax, [0x20001234]` have different bytes.

The answer is **relocation masking**. When a PE binary (DLL/EXE) is loaded into memory, the OS patches absolute addresses through a relocation table. Our comparison tool reads this table and zeroes out those patched bytes in both the retail and rebuilt versions before comparing. So as long as the *instructions* are the same and the *relocations* point to corresponding data, it's a match.

This is what makes simple accessor functions so reliable. `StaticClass()` always generates `mov eax, [relocation]; ret` — the opcode bytes are identical; only the address inside the relocation differs.

## What's Actually in the 1,213?

Breaking it down by function type:

- **StaticClass()** — ~120 functions, 6 bytes each. Every class gets one.
- **operator new()** (allocating) — ~100 functions, 44 bytes each. The `StaticAllocateObject` wrapper.
- **operator new()** (placement) — ~50 functions, 5 bytes each. Just `return (void*)Mem`.
- **InternalConstructor()** — ~100 functions, 14 bytes each. Placement-new into preallocated memory.
- **Destructors** — various sizes. The compiler-generated `vftable` teardown chains.
- **Copy constructors and operator=** — larger functions that happen to compile identically.
- **Misc small functions** — constructors, simple member accessors, trivial stubs.

The Engine.dll contributed the most raw numbers (569 functions), which makes sense — it defines the most classes.

## The Lesson

The real lesson here isn't about macro-generated boilerplate. It's about **measurement blind spots**. We had over a thousand functions producing perfect output — proof that our build system, struct layouts, and compiler configuration were fundamentally sound for large swathes of the codebase.

But because we only measured annotated functions, our metrics told a much gloomier story. It's like testing 5% of your product and concluding the whole thing is broken. Sometimes the fix isn't writing better code — it's expanding what you measure.
